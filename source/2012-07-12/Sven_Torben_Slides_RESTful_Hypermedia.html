Architectural Style

[...] the Representational State Transfer (REST) [is an] architectural style for
distributed hypermedia systems.
REST provides a set of architectural constraints that, when applied as a whole, emphasizes scalability of component interactions, generality of interfaces, independent deployment of components, and intermediary components to reduce interaction latency, enforce security, and encapsulate legacy systems.
(Reference: Fielding's dissertation)

Hypermedia Factors

introduced by Mike Amundsen

Links

  • Embed Links
  • Outbound Links
  • Templated Links
  • Idempotent Links
  • Non-Idempotent Links

Embed Links (LE)

Transclusion

  • Client should dereference URI by read operation
  • Display response in current output window
  • Merge current content with response content
HTML 
<img src="..." />
XInclude 
<x:include href="..." />

Outbound Links (LO)

Traversal or navigational link

  • Client should dereference URI by read operation
  • Treat response as complete display
  • (Metadata: Replace current display or display new viewport)
HTML 
<a href="...">...</a>
<a href="..." target="_blank">...</a>

Templated Links (LT)

URI Templates

  • Indicates parameters for read operation
  • Construction of URIs depends on documentation of media-type
HTML 
<form method="get" action="./search">
    <input type="text" name="query" value="" />
    <input type="submit" />
</form>
Custom 
<link href="./search?query={query}" />

Idempotent Links (LI)

Idempotent Updates

  • Support for idempotent submissions to a server

Atom media type 
<link rel="edit" href="http://example.org/items/1" />
HTML with code-on-demand 
<script type="text/javascript">
function delete(id) {
    var client = new XmlHttpRequest();
    client.open("DELETE", "http://example.org/items/" + id);
}
</script>

Non-Idempotent Links (LN)

Non-Idempotent Submits

  • Template holding elements/hints for the client
  • Construction of message body based on media-types' rules
HTML 
<form method="post" action="...">
    <textarea name="comment"></textarea>
    <input type="submit" />
</form>

Control Data

  • Read Controls
  • Update Controls
  • Method Controls
  • Link Annotation Controls

Read Controls (CR)

  • Expose control information to clients
  • Supports manipulation of control data for read operations
XInclude 
<x:include href="..." accept-language="de, en;q0.5" />

Update Controls (CU)

  • Supports manipulation of control data for send/update operations
HTML 
<form method="post" action="..." enctype="text/plain">
    <textarea name="comment"></textarea>
    <input type="submit" />
</form>

Method Controls (CM)

  • Supports selection of application protocol's
    method used for request
  • Construction of URIs depends on documentation
    of media-type
HTML 
<form method="post" action="...">
    <!-- send operation -->
</form>

<form method="get" action="...">
    <!-- read operation -->
</form>

Link Annotation Controls

  • Define inline metadata for links themselves
  • Locate and understand meaning of a link
  • Needs an agreed-upon set of keywords
HTML 
<link rel="stylesheet" href="...">
Atom media type 
<link rel="edit" href="...">

application/json ?

Hypermedia Design Elements

  • Base Format
    • HTML
    • XML
    • JSON
    • Binary ...
  • State Transfer
    • None (Read only)
    • Predefined (via external documentation)
    • Ad-Hoc (in-message hypermedia-controls)
  • Design Style (domain affinity)
    • Specific
    • General
    • Agnostic
  • Applicataion Flow
    • None (e.g. text/uri-list)
    • Intrinsic (e.g. Atom Pub)
    • Applied (e.g. HTML with profile meta-tag)
